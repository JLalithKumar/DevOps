# Exercise 14 – Distributed Tracing Investigation

## Objective

The objective of this exercise is to understand how Distributed Tracing helps identify latency issues in a microservices architecture.

Unlike logs and metrics, distributed tracing provides complete visibility into how a request travels through multiple services and identifies exactly where time is being spent.

---

# Incident

Users reported that the Checkout API had become slow.

### User Complaint

```
Checkout API is slow.
```

Available monitoring information:

```
Grafana
95th Percentile Latency:
4.8 Seconds

Prometheus
Request Count:
Normal
```

Distributed Trace:

```
Checkout Service
        │
        ▼
Inventory Service
        │
        ▼
Payment Service
```

The objective was to determine which service was responsible for the increased response time.

---

# Architecture

The application consists of three Flask microservices.

```
                Client
                   │
                   ▼
          Checkout Service
                   │
                   ▼
         Inventory Service
                   │
                   ▼
          Payment Service
```

The services communicate through HTTP REST APIs.

OpenTelemetry instruments each service and exports traces to Tempo.

Grafana is used to visualize distributed traces.

---

# Technology Stack

- Docker
- Docker Compose
- Python Flask
- OpenTelemetry
- Grafana Tempo

# Components

## Checkout Service

Responsible for receiving customer requests.

Calls the Inventory Service to continue processing the order.

Endpoint:

```
GET /checkout
```

---

## Inventory Service

Responsible for inventory processing.

Receives requests from Checkout Service and forwards them to Payment Service.

Endpoint:

```
GET /inventory
```

---

## Payment Service

Responsible for payment processing.

To simulate a production latency issue, an artificial delay was introduced.

```python
time.sleep(4)
```

This delay represents a slow payment gateway or an external payment provider.

Endpoint:

```
GET /payment
```

# Trace Analysis

Observed durations:

```
Checkout Service
≈ 4.02 seconds

Inventory Service
≈ 4.01 seconds

Payment Service
≈ 4.00 seconds
```

Although Checkout and Inventory also appeared slow, the trace showed that both services were waiting for the Payment Service to complete.

The Payment Service consumed almost the entire request duration.

---

# Root Cause Analysis

## Symptoms

- Checkout API slow
- High request latency
- No application failures
- No infrastructure issues

---

## Metrics

Prometheus indicated that request count remained normal.

There was no evidence of excessive traffic or resource exhaustion.

---

## Logs

Application logs showed successful HTTP 200 responses.

No errors or exceptions were found.

---

## Traces

Tempo waterfall clearly identified the bottleneck.

```
Checkout
        │
        ▼
Inventory
        │
        ▼
Payment
```

The Payment Service consistently required approximately four seconds to complete.

---

# Root Cause

The latency originated from the Payment Service.

The application intentionally simulated a slow payment gateway using:

```python
time.sleep(4)
```

Checkout and Inventory were blocked while waiting for the Payment Service to return a response.

---

# Immediate Resolution

Remove or optimize the slow operation inside the Payment Service.

Possible production fixes include:

- Optimize database queries
- Reduce external API latency
- Improve payment gateway performance
- Implement caching where appropriate
- Introduce asynchronous processing if applicable

---

# Long-Term Prevention

- Monitor service latency
- Create Grafana dashboards
- Configure latency alerts
- Monitor p95 and p99 latency
- Implement Service Level Objectives (SLOs)
- Continuously analyze distributed traces

---

# Key Learning

Distributed Tracing enables engineers to identify exactly where request latency occurs across multiple services.

Unlike metrics, which indicate that latency exists, and logs, which report application events, traces provide complete visibility into the execution path of every request.

This exercise demonstrated how Tempo and Grafana can be used together to quickly identify the true bottleneck in a distributed system.

---

# Conclusion

The investigation confirmed that the Checkout Service itself was functioning correctly.

The increased response time was caused by the Payment Service, which intentionally introduced a four-second delay.

Using Distributed Tracing, the latency was isolated to the correct service without relying on guesswork, demonstrating the value of observability in modern microservices architectures.
